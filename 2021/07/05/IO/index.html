IO底层

IO读写基本原理

操作系统将内存（虚拟内存）划分为两部分：一部分是内核空间，另一部分是用户空间。操作系统内核程序既有权限访问受保护的内核空间，内核空间总是驻留在内存，为操作系统内核保留。每个应用程序进程都有一个单独的用户空间，对应的进程处于用户态。用户态进程不能访问内核空间中的数据，也不能直接调用内核函数，需要将进程切换到内核态才能进行系统调用。

应用程序的IO操作实际上不是物理设别级别的读写，而是缓存的复制。把数据在内核缓冲区和应用程序的进程缓冲区之间复制。底层的读写交换操作由操作系统内核完成。

四种主要IO模型

阻塞：
内核IO操作彻底完成后才返回到用户空间执行用户程序，用户进程一直等待，而不能做别的事情
非阻塞：
用户进程获得内核返回的状态值就返回自己的空间，可以去做别的事情
同步：
用户空间是主动发起IO请求的一方
异步：
系统内核是主动发起IO请求的一方

同步阻塞IO

用户空间主动发起，需要等待内核IO操作彻底完成后才返回到用户空间

同步非阻塞IO

用户空间的程序不需要等待内核IO操作彻底完成，可以立刻返回用户空间去执行后续的指令，与此同时，内核会立即返回用户一个IO状态值

IO多路复用（select/epoll系统调用）

一个用户进程可以监视多个文件描述符，一旦某个描述符就绪，内核就能够将文件描述符的就绪状态返回给用户进程，用户空间可以根据文件描述符的就绪状态进行相应的IO系统调用

异步IO

用户空间的线程变成被动接受者，而内核空间成为主动调用者。在异步IO模型中，当用户线程收到通知，数据v已经被内核读取完毕并放在用户缓冲区，内核在IO完成后通知用户线程直接使用

java NIO

Channel、Buffer、Selector

OIO

面向流，阻塞，没有选择器的概念

NIO

面向缓冲区，非阻塞，有选择器

通道

所有NIO的IO操作都是通过链接通道完成。一个通道类似OIO中两个流的结合体，既可以从通道读取数据v，也可以向通道写入数据。

FileChannel

文件通道，用于文件的数据读写

SocketChannel

套接字通道，用于套接字TCP连接的数据读写

ServerSocketChannel

服务器套接字通道(或服务器监听通道),监听TCP连接请求，为每个监听到的请求创建一个SocketChannel

DategramChannel

数据报通道，用于UDP的数据读写

选择器

选择器的使命是完成IO的多路复用，其主要工作是通道的注册、监听、事件查询。选择器和通道的关系是监控和被监控的关系。
可以理解为一个IO事件的监听与查询器。通过选择器，一个线程可以查询多个通道的IO事件的就绪状态

IO事件

通道处于某个IO操作的就绪状态，表示通道具备执行某个IO操作的条件
可供选择器监控的通道IO时间类型

  • 可读：SelectionKey.OP_READ
  • 可写：SelectionKEY.OP_WRITE
  • 连接：SelectionKey.OP_CONNECT
  • 接收：SelectionKey.OP_ACCEPT

SelectionKey

一旦在通道中发生了某些IO事件(就绪状态达成),并且是在选择器中注册过的IO事件，就会被选择器选中，并放入SelectionKey的集合中。
SelectionKey就是那些被选择器选中的IO事件。

缓冲区

通道的读取，就是将数据从通道读取到缓冲区;通道的写入，就是将数据从缓冲区写入通道

Reactor模式

Reactor:负责查询IO事件,当检测到一个IO事件时将其发送给相应的Handler处理器去处理
Handler:与IO事件绑定，负责IO事件的处理，完成真正的连接建立、通道的读取、处理业务逻辑、负责将结果写到通道
